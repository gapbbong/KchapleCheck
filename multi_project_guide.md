# 🚀 단일 Supabase에서 여러 프로젝트 관리하기 (NUC/로컬 서버용)

사용 중인 NUC(8GB~16GB RAM) 사양에서는 Supabase 엔진을 여러 개 띄우는 것보다, **하나의 엔진을 똑똑하게 나누어 쓰는 것**이 하드웨어 자원을 아끼는 가장 좋은 방법입니다.

---

## 1. 프로젝트 접속 정보 재사용
모든 앱(예바 출석, 퀴즈 앱, 급식 앱 등)에서 **동일한 `SUPABASE_URL`과 `SUPABASE_ANON_KEY`**를 그대로 사용하시면 됩니다. 이렇게 하면 RAM 소모가 `2GB` 선에서 멈추므로 매우 경제적입니다.

- **`config.js` 공유**: 각 앱 폴더마다 `config.js`를 만들되, 내용은 동일하게 유지하세요.
- **ANON_KEY 보안**: 앱 안에서 `ANON_KEY`를 사용하더라도 테이블마다 보안 규칙(RLS)을 잘 설정해두면 서로의 데이터를 함부로 건드리지 못하게 막을 수 있습니다.

---

## 2. 데이터 격리 전략 (2가지 방법)

### 방법 A: 테이블 이름 접두어 (Table Prefix) - 추천 ★★★
가장 직관적이고 쉬운 방법입니다. 저와 함께 `kchaple_`을 붙인 방식입니다.

- **규칙**: `[앱이름]_[테이블명]`
  - 앱 1: `kchaple_students`, `kchaple_attendance`
  - 앱 2: `quiz_scores`, `quiz_questions`
  - 앱 3: `dinner_menu`, `dinner_votes`
- **장점**: SQL Editor나 Table Editor에서 프로젝트별로 테이블이 가나다순으로 모여 있어 관리가 편합니다.

### 방법 B: 데이터베이스 스키마 (PostgreSQL Schema) - 중급 ★
윈도우의 '폴더' 개념처럼 데이터베이스 안에 논리적인 구역을 완전히 나누는 방식입니다.

- **방법**: SQL Editor에서 `CREATE SCHEMA app_quiz;` 명령어로 폴더를 생성합니다.
- **사용**: `app_quiz.students` 처럼 앞에 스키마 명을 붙여 호출합니다.
- **장점**: `public` 스키마가 복잡해지지 않고 깔끔합니다. 
- **주의**: Supabase API(supabase-js)에서 기본적으로 `public` 스키마만 바라보므로 하단과 같은 추가 설정이 필요합니다.
  ```javascript
  const supabase = createClient(URL, KEY, { db: { schema: 'app_quiz' } });
  ```

---

## 3. 서비스 롤 키 (SERVICE_ROLE_KEY) 활용법
`SERVICE_ROLE_KEY`는 **"모든 보안(RLS)을 무시하는 마스터 키"**입니다. 

- **활용**: 
  - 앱(클라이언트) 코드에는 절대 넣지 마세요! (누구나 DB를 다 지울 수 있게 됩니다.)
  - 마이그레이션 도구(`import_data.html`)처럼 **데이터를 대량으로 밀어넣거나 관리할 때만** 잠깐 가져와서 입력하는 방식으로 사용하세요.
- **연동**: 다른 앱을 개발할 때도 관리가 필요하면 해당 앱 전용 마이그레이션 도구를 만들어 이 키를 활용하시면 됩니다.

---

## 💡 결론 요약
1. **서버(IP:8000)**와 **개발 키**는 하나로 계속 쓰세요 (RAM 아끼기).
2. 새로운 기능을 만들 땐 항상 **`프로젝트명_`** 을 붙여서 테이블을 만드세요.
3. 각 테이블마다 반드시 **RLS(보안 정책)**를 설정해서 `ANON_KEY`로도 안전하게 통신하게 하세요.
